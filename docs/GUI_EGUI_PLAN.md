# План Rust GUI на egui/eframe

GUI нужен не как украшение поверх движка, а как отдельный модуль для игры, отладки и экспериментов с поиском. При этом GUI не должен смешиваться с ядром правил и поиском.

Выбранное направление: `egui/eframe`. Это не нативный интерфейс в смысле системных виджетов, но он подходит для экспериментов на Rust: быстро рисует окна, панели, доску и настройки, собирается в обычное desktop-приложение и не требует уходить из Rust-экосистемы.

## Главная архитектурная граница

Первый вариант GUI должен общаться с движком как с UCI-процессом:

```text
GUI на egui/eframe  <->  UCI stdin/stdout  <->  engine
```

Такой подход даёт несколько плюсов:

- GUI не ломает ядро;
- движок остаётся совместимым с внешними шахматными GUI;
- ошибки интерфейса не портят модель позиции внутри движка;
- можно отдельно тестировать UCI;
- можно заменить GUI, не переписывая поиск.

Позже можно добавить in-process режим, где GUI подключает движок как библиотеку. Но это второй этап, а не первый.

## MVP интерфейса

Минимальный GUI должен уметь:

- показывать шахматную доску;
- делать ход кликом или drag-and-drop;
- подсвечивать выбранную фигуру и доступные легальные ходы;
- запускать новую партию;
- выбирать сторону игрока;
- переворачивать доску;
- отправлять позицию движку через UCI;
- получать `bestmove`;
- показывать список ходов;
- принимать FEN вручную;
- задавать глубину поиска;
- отображать статус: чей ход, думает ли движок, последний найденный ход.

Это достаточный первый GUI. Всё остальное лучше добавлять после того, как этот слой станет стабильным.

## Отладочные панели

Главная ценность собственного GUI — не только играть партии, но и видеть внутренности движка. Поэтому после MVP нужны отладочные панели:

- текущий FEN;
- список легальных ходов;
- результат `perft` для малых глубин;
- лучший ход и оценка;
- principal variation, когда поиск начнёт её возвращать;
- глубина поиска;
- число просмотренных узлов;
- время поиска;
- выбранная методика поиска;
- активный профиль стиля.

Это сделает GUI инструментом разработки, а не просто шахматной доской.

## Настройки характера игры

Когда в движке появятся параметры стиля, GUI должен дать к ним прямой доступ. Не через скрытый конфиг, а через понятные ползунки или переключатели.

Возможные настройки:

```text
Агрессия             0..100
Осторожность         0..100
Материализм          0..100
Теоретичность        0..100
Риск                 0..100
Любовь к открытым позициям   0..100
Любовь к закрытым позициям   0..100
Эндшпильная подготовка       0..100
Миттельшпильное давление     0..100
Скорость игры        0..100
```

Эти настройки не должны быть косметикой. Каждая настройка обязана иметь понятное влияние на оценку позиции, сортировку ходов или управление временем.

## Методики поиска

GUI должен позволять выбирать и сравнивать методики поиска. Например:

- `classic-alpha-beta`;
- `tactical-alpha-beta`;
- `positional-alpha-beta`;
- `endgame-biased`;
- `hybrid-controller`.

Пока таких методик нет, GUI может показывать только текущий режим. Но архитектурно место под выбор методики лучше заложить рано.

## Рекомендуемая структура проекта

На раннем этапе можно оставить один crate и добавить GUI как feature. Но более чистый путь — постепенно прийти к workspace:

```text
rchess/
  Cargo.toml
  crates/
    rchess_core/
      src/lib.rs
    rchess_engine/
      src/main.rs
    rchess_gui/
      src/main.rs
```

Где:

- `rchess_core` хранит правила, позицию, ходы, FEN, perft;
- `rchess_engine` содержит поиск и UCI-бинарник;
- `rchess_gui` содержит `egui/eframe`-интерфейс;
- GUI общается с engine через UCI или, позже, через библиотечный API.

Для первого GUI-патча можно не делать полный workspace сразу. Допустим промежуточный вариант:

```text
src/
  chess.rs
  search.rs
  uci.rs
  main.rs
  gui.rs        # только за feature = "gui"
```

Но если GUI начнёт расти, его нужно вынести отдельно.

## Что нельзя допускать

- GUI не должен напрямую менять внутренние поля позиции в обход правил.
- GUI не должен знать детали alpha-beta поиска.
- Поиск не должен импортировать `egui`.
- UCI не должен превращаться в свалку GUI-команд.
- Настройки стиля не должны быть случайными флагами без документации.

## Первый технический GUI-патч

Разумный следующий шаг после этого документа:

1. Добавить feature `gui`.
2. Добавить зависимости `eframe` и `egui` только под GUI.
3. Создать простое окно с доской.
4. Научить GUI запускать текущий бинарник движка как UCI-процесс.
5. Реализовать `uci`, `isready`, `position startpos moves ...`, `go depth N`.
6. Отрисовать ответ `bestmove` на доске.
7. Не трогать логику `chess.rs` и `search.rs`, если для GUI это не требуется.

Так проект сохранит чистую границу: движок остаётся движком, GUI остаётся клиентом.

## Текущий MVP

Первый GUI-MVP вынесен в отдельный бинарник:

```text
src/bin/rchess-gui.rs
```

Запуск:

```bash
cargo run --bin rchess-gui
```

Подробности текущего MVP описаны в [`GUI_MVP.md`](GUI_MVP.md).


## PGN

PGN должен оставаться не GUI-специфичной возможностью. GUI вызывает общий модуль `src/pgn.rs`, а не держит собственный парсер. Это важно, чтобы CLI, тесты и будущие инструменты анализа партии использовали одну и ту же проверку ходов.

Текстовый импорт/экспорт уже есть. Следующий уровень добавил копирование PGN, открытие/сохранение через ручное поле пути и список ходов в SAN. Нативный файловый диалог остаётся отдельной будущей задачей.


## External engines and engine-vs-engine

GUI уже умеет выбирать backend-движок. Это важно для будущих сравнений: интерфейс не должен считать, что существует только один движок.

Текущие backend-режимы:

```text
rchess internal UCI
Stockfish 10
Custom UCI executable
```

Stockfish 10 подключён как внешний UCI-ориентир. Он не становится частью Rust-ядра и не меняет принцип отказа от NNUE/ИИ внутри собственного движка.

Будущий `engine vs engine` режим лучше делать не как две кнопки в текущей панели, а как отдельный контроллер партии:

```text
White UCI process  <->  Game controller  <->  Black UCI process
```

Контроллер должен владеть состоянием партии, временем, PGN, результатом и проверкой легальности. Каждый движок должен оставаться заменяемым UCI-процессом.
